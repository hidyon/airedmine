# Seed Scenario Design

AIRedmine の seed データは、単に issue を多く持つ Redmine ではなく、AI エージェントが「実プロジェクトの状態」を読み解くためのデモ素材として扱う。
このドキュメントは M18 の seed 改善で追加・調整するシナリオを定義する。

## 現状

既存 seed は `kintai-next` という次世代勤怠管理 SaaS プロジェクトを作成する。

- Sprint 1〜4、status、priority、tracker、6 名のメンバーが定義済み。
- `scripts/redmine/seed-data/*.yml` に 510 件の issue がある。
- 鈴木（frontend）、田中（lead）、佐藤（backend）、伊藤（infra）、山田（qa）、中村（pm）の担当別ファイルに分かれている。
- 鈴木担当 issue にはコメント履歴が多く、仕様判断、レビュー、パフォーマンス議論の素材がある。
- seed runner は issue 作成と journals には対応しているが、issue relation はまだ作成していない。

主な不足:

- 510 件はあるが、エピックや依存関係として読み取れる明示的なまとまりが弱い。
- PM Dashboard が見せたい「遅延リスク」「担当者偏り」「PM 判断待ち」が偶然に依存している。
- Chat デモで使う代表質問と、期待される観察ポイントが README から辿れない。
- 再投入時の重複・リセット方針が M18 のデモ運用として明文化されていない。

## 目指すデモ体験

### 開発者

- 「今日何からやればいい？」で、自分の担当 issue の中からブロッカー、期限、優先度、更新停滞を踏まえた候補を受け取れる。
- issue 詳細やコメント履歴から、仕様変更・レビュー指摘・未回答質問を要約できる。
- 意味検索で「パフォーマンス」「承認フロー」「セキュリティ」などの概念から関連 issue を見つけられる。

### PM

- Dashboard で Sprint 3 の遅延、担当者負荷、期限切れ、優先度偏りを一目で確認できる。
- Chat で「PM 判断待ちは？」「リリースリスクは？」と聞いたとき、実際の issue とコメント履歴に基づく回答が返る。
- 一部 issue が Feedback / In Progress で停滞しており、次に誰へ確認すべきか分かる。

### Audit / Proposal

- seed 投入後に Chat から更新提案を作り、Audit に履歴が残る流れを README の質問例から試せる。
- 危険操作や失敗ログの説明に使える issue がある。

## プロジェクトシナリオ

### Scenario A: Sprint 3 リリースゲート

目的: PM Dashboard と Chat で「今リリースできるか」を判断する素材にする。

含める状態:

- Sprint 3 の Urgent / High issue が複数残っている。
- QA の最終確認、セキュリティ確認、PM 判断待ちが混在する。
- 一部は Resolved だが、確認コメント待ちで Closed になっていない。

代表 issue 候補:

- 山田: `本番リリース前の最終確認テストを実施する`
- 山田: `認証バイパスの試みを検証する`
- 中村: `PM判断待ち: v1.0 のリリース日程を最終確定する`
- 田中: `品質基準を満たしているかのゲートレビューを行う`

後続作業:

- ISS-095 で上記候補に判断履歴コメントを追加する。
- 「リリース判断に残っているリスクは？」のデモ質問に対応する。

### Scenario B: 申請・承認フローの仕様揺れ

目的: Chat がコメント履歴から意思決定・未決事項を要約できることを示す。

含める状態:

- フロント、バックエンド、QA、PM にまたがる承認フロー issue を配置する。
- 仕様変更、代理承認、期間指定、一括承認、リアルタイム反映の議論をコメントに残す。
- 一部 issue は PM 判断待ち、または backend API 待ちとして停滞させる。

代表 issue 候補:

- 鈴木: `代理承認ユーザーの選択 UI を実装する`
- 鈴木: `一括承認機能の UI を実装する`
- 佐藤: `一括承認 API エンドポイントを実装する`
- 山田: `一括承認で一部が承認されないバグを報告する`
- 中村: `PM判断待ち: 打刻修正の申請期限（何日以内まで？）を決める`

後続作業:

- ISS-095 で関係する issue に共通語彙（代理承認、一括承認、申請期限）を増やす。
- seed runner が relation に対応する場合は blocks / relates を付ける。

### Scenario C: レポート性能と大量データ対応

目的: M19 のパフォーマンス分析と、意味検索デモの橋渡しにする。

含める状態:

- 部署別レポート、月次カレンダー、PDF 一括生成、DB 遅延など、フロント・バック・インフラ・QA の性能 issue を横断させる。
- 数値（340ms、3 秒以内、500 人規模、Lighthouse 72 など）をコメントに含める。
- 一部 issue は Sprint 3 の高優先度として PM Dashboard に現れるようにする。

代表 issue 候補:

- 鈴木: `部署別勤怠集計テーブルを実装する`
- 鈴木: `月次勤怠カレンダーの初期描画パフォーマンスを改善する`
- 鈴木: `モバイル向けの Lighthouse スコアを 90 点以上に改善する`
- 伊藤: `PostgreSQL のスロークエリログを有効化して分析する`
- 山田: `レポート生成 API の応答時間を計測する`

後続作業:

- ISS-095 で性能 issue のコメントを増やし、M19 の計測対象と語彙を合わせる。
- README のデモ質問に「パフォーマンスが遅い原因候補は？」を追加する。

### Scenario D: 担当者負荷と停滞

目的: PM Dashboard の担当者別負荷と停滞 issue を自然に見せる。

含める状態:

- 鈴木と山田に Sprint 3 の open issue が多い状態を維持する。
- 中村に PM 判断待ち issue を複数持たせる。
- 10 日以上更新されていない Feedback / In Progress issue を各ロールに散らす。

代表 issue 候補:

- 中村: `PM判断待ち:*` の各 issue
- 鈴木: Feedback の UI issue
- 伊藤: `ペネトレーションテストの結果を対応する`
- 山田: `月初に集計がリセットされないバグを報告する`

後続作業:

- ISS-095 で `updated_days_ago` を意図的に調整し、停滞として見える issue を明示する。
- ISS-097 で PM Dashboard の見どころとして担当者負荷・停滞を案内する。

### Scenario E: Redmine 更新提案のデモ

目的: Chat から proposal を作り、確認後に Audit で履歴を見る流れを説明する。

含める状態:

- コメント追加に向いた issue（レビュー結果や確認メモを残す）。
- 期日変更に向いた issue（Sprint 4 へ延期する判断）。
- 優先度変更に向いた issue（Urgent に上げる理由がある）。
- ステータス変更に向いた issue（Resolved から Closed にする候補）。

後続作業:

- ISS-097 で「#NNN にコメント」「期日を変更」「優先度を High に」などの質問例を README に追加する。
- issue 番号は seed 投入時に固定しづらいため、README では件名検索から始める例にする。

## 代表質問

開発者向け:

- 「私の今日の issue を優先順に教えて」
- 「承認フローまわりでブロッカーになっている issue は？」
- 「月次カレンダーのパフォーマンス改善について、これまでの議論を要約して」
- 「パフォーマンスが遅いという相談に関係する issue を探して」

PM 向け:

- 「Sprint 3 のリリース判断で残っているリスクは？」
- 「PM 判断待ちの issue をまとめて」
- 「担当者ごとの負荷で偏っているところは？」
- 「期限切れ・停滞・Urgent をまとめて次の定例アジェンダにして」

更新提案向け:

- 「承認フローの仕様確認が必要な issue に、PM 確認待ちとコメントして」
- 「リリース日程が未確定の issue の優先度を High にする提案を作って」
- 「Resolved の QA 確認済み issue を Closed にする候補を教えて」

## 後続 issue への接続

- ISS-095: この設計に沿って seed issue、コメント履歴、ステータス、優先度、更新日、必要なら relation を拡充する。
- ISS-096: 追加データを何度も投入できるように、seed 再投入・リセット方針を確認する。
- ISS-097: 代表質問と観察ポイントを README に追加する。
- ISS-098〜101: Scenario C の性能系データを使って、主要 API と画面の計測を行う。
