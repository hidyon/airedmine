# AIRedmine 開発ルール

このファイルは AIRedmine をユーザーと Claude が一緒に開発するための運用ルールを定義する。
作業前に必ず読み、進め方に迷ったときはこのファイルを優先する。

## 重要なファイル

- `README.md`: アプリの目的、起動方法、利用方法、現在の仕様。
- `docs/roadmap.md`: 現在進行中の開発ロードマップ、優先順位、マイルストーン。
- `docs/roadmaplog.md`: 終了したマイルストーンと過去の変更履歴。
- `docs/issues.md`: 取り組む issue、進行中 issue、完了 issue。
- `docs/issueslog.md`: issue の検討ログ、判断理由、保留事項、代替案。

ユーザーが `readme.md` と書いた場合は `README.md` を指す。
ユーザーが `issue検討ログ` または `issueslog.md` と書いた場合は `docs/issueslog.md` を指す。

## 開発ループ

1. ロードマップの策定
2. 取り組む issue を決定
3. issue の解決

各ループで `docs/roadmap.md` と `docs/issues.md` に現在の判断が反映されているか確認する。
実装だけを先行させず、何をなぜ作るのかを記録してから進める。

## ロードマップの策定

- アプリの目的、利用者、主要なユースケースを確認する。
- 直近で価値が出る単位に分割する。
- 各項目に優先度、期待成果、完了条件を置く。
- 各マイルストーンには対応する issue と状態を記載する。
- ロードマップを変更した場合は、現在進行中の内容を `docs/roadmap.md` に、終了済みの履歴を `docs/roadmaplog.md` に残す。

## 取り組む Issue の決定

- `docs/issues.md` から次に取り組む issue を選ぶ。
- 優先度、依存関係、影響範囲、検証しやすさを見て決める。
- 迷う場合は `docs/issueslog.md` に比較検討を残す。
- 取り組む issue は開始時に状態を `In Progress` にする。

## Issue 解決の手順

issue はスペック駆動開発で解決する。
実装前に少なくとも次の 3 点を数行で会話上に共有する。

- 何を解くか。
- どの画面、API、docs、データを変えるか。
- 何を確認できれば完了とするか。

小さな修正でも「この前提で進める」と明示してから編集する。
緊急の不具合修正で先に手を動かす場合も、最初の作業報告で仮仕様と検証方針を共有する。

### 仕様の構成

1. **要求仕様**: 誰のどんな問題を解くか。完了時に何ができるか。対象外は何か。
2. **機能仕様**: UI、API、データ、状態、エラー処理の振る舞い。既存機能への影響。Redmine API 連携がある場合は利用エンドポイント・入力・出力・失敗時の扱い。
3. **テスト仕様**: 何を確認すれば完了か。自動/手動/API/ブラウザのどれで検証するか。失敗時に見るべきログや画面状態。

### 実施順

1. 実装
2. テスト
3. テスト結果を確認
4. issue クローズの判定

クローズ判定では要求・機能・テスト仕様のすべてを満たしているか確認する。
満たしていない場合は issue を閉じず、残作業を `docs/issues.md` に記録する。

クローズ前に以下も確認する。

- 削除した機能・ルートに対応するファイル（View、コンポーネント等）が残っていないか。
- 新しく追加した環境変数が `.env.example` と `docker-compose.yml` の両方に反映されているか。
- アーキテクチャや起動構成を変えた場合、devcontainer が正しく起動するか確認したか。
- ユーザー・認証・ロール絡みの変更は、別ユーザーでログインして動作を確認したか。

## Git 運用

- issue をクローズできる状態になったらコミットする。
- コミット前に関連ファイルの差分を確認する。
- コミットメッセージには issue 番号または issue 名が分かる内容を入れる。
- コミット後、リモートが設定されていて push 可能なら push する。
- push できない場合は理由をユーザーに共有し、次の操作を提案する。

## 自己改善

- 定期的に進め方を振り返る。
- 開発ループ、issue 解決手順、仕様の書き方、テスト方法に改善余地があれば提案する。
- `CLAUDE.md` の改善が適切な場合は変更案をユーザーに提案する。
- Docker や外部サービス接続で得た手順は README または docs に再現可能な形で残す。
- issue 一覧は Status と見出しが矛盾しないよう定期的に整理する。

## 作業姿勢

- 既存の設計、命名、ファイル構成を尊重する。
- 変更は issue の範囲に収める。
- 実装前に必要な仕様を明確にし、会話上で短く共有してから着手する。
- 実装後に検証結果を共有する。
- ユーザーの未コミット変更を勝手に戻さない。
- 不明点があっても合理的に仮定できる場合は前に進め、リスクが高い場合だけ確認する。
