# AIRedmine 開発ルール

このファイルは、AIRedmine をユーザーと Codex が一緒に開発するための運用ルールを定義する。
作業前に必ず読み、進め方に迷ったときはこのファイルを優先する。

## 重要なファイル

- `README.md`: アプリの目的、起動方法、利用方法、現在の仕様を説明する。
- `docs/roadmap.md`: 現在進行中の開発ロードマップ、優先順位、マイルストーンを管理する。
- `docs/roadmaplog.md`: 終了したマイルストーンと過去の変更履歴を保管する。
- `docs/issues.md`: 取り組む issue、進行中 issue、完了 issue を管理する。
- `docs/issueslog.md`: issue の検討ログ、判断理由、保留事項、代替案を記録する。

ユーザーが `readme.md` と書いた場合は `README.md` を指すものとして扱う。
ユーザーが `issue検討ログ` または `issueslog.md` と書いた場合は `docs/issueslog.md` を指す。

## 開発ループ

開発は次のループで進める。

1. ロードマップの策定
2. 取り組む issue を決定
3. issue の解決

各ループでは、現在の判断が `docs/roadmap.md` と `docs/issues.md` に反映されているか確認する。
実装だけを先行させず、何をなぜ作るのかを記録してから進める。

## ロードマップの策定

- アプリの目的、利用者、主要なユースケースを確認する。
- 直近で価値が出る単位に分割する。
- 各項目に優先度、期待成果、完了条件を置く。
- 各マイルストーンには、対応する issue と状態を記載する。
- ロードマップを変更した場合は、現在進行中の内容を `docs/roadmap.md` に、終了済みの履歴を `docs/roadmaplog.md` に残す。

## 取り組む Issue の決定

- `docs/issues.md` から次に取り組む issue を選ぶ。
- 優先度、依存関係、影響範囲、検証しやすさを見て決める。
- 迷う場合は `docs/issueslog.md` に比較検討を残す。
- 取り組む issue は、開始時に状態を `In Progress` にする。

## Issue 解決の手順

issue はスペック駆動開発で解決する。
実装前に、少なくとも次の仕様を決める。
実装に入る前に、Codex は決めた仕様を会話上で短く共有する。
共有は長い設計書ではなく、次の 3 点を数行で明示する。

- 何を解くか。
- どの画面、API、docs、データを変えるか。
- 何を確認できれば完了とするか。

小さな修正でも、最低限「この前提で進める」と明示してから編集する。
緊急の不具合修正などで先に手を動かす必要がある場合も、最初の作業報告で仮仕様と検証方針を共有する。

1. 要求仕様
   - 誰のどんな問題を解くのか。
   - 完了時に何ができるようになるのか。
   - 対象外にすることは何か。

2. 機能仕様
   - UI、API、データ、状態、エラー処理の振る舞い。
   - 既存機能への影響。
   - Redmine API 連携がある場合は利用エンドポイント、入力、出力、失敗時の扱い。

3. テスト仕様
   - 何を確認すれば完了と言えるか。
   - 自動テスト、手動テスト、API 確認、ブラウザ確認のどれで検証するか。
   - 失敗時に見るべきログや画面状態。

その後、次の順で進める。

1. 実装
2. テスト
3. テスト結果を確認
4. issue クローズの判定

クローズ判定では、要求仕様、機能仕様、テスト仕様のすべてを満たしているか確認する。
満たしていない場合は issue を閉じず、残作業を `docs/issues.md` に記録する。

クローズ前に以下も確認する。

- 削除した機能・ルートに対応するファイル（View、コンポーネント等）が残っていないか。
- 新しく追加した環境変数が `.env.example` と `docker-compose.yml` の両方に反映されているか。
- アーキテクチャや起動構成を変えた場合、devcontainer が正しく起動するか確認したか。
- ユーザー・認証・ロール絡みの変更は、別ユーザーでログインして動作を確認したか。

## Git 運用

- issue をクローズできる状態になったらコミットする。
- コミット前に関連ファイルの差分を確認する。
- コミットメッセージには issue 番号または issue 名が分かる内容を入れる。
- コミット後、リモートが設定されていて push 可能なら push する。
- push できない場合は理由をユーザーに共有し、次の操作を提案する。

## 自己改善

- 定期的に進め方を振り返る。
- 開発ループ、issue 解決手順、仕様の書き方、テスト方法に改善余地があれば提案する。
- `agent.md` の改善が適切な場合は、変更案をユーザーに提案する。
- `agent.md` 以外の改善方法があれば、それも提案する。
- 例: issue テンプレート、テストチェックリスト、Redmine API 調査メモ、リリースノート、開発コマンド一覧。
- Docker や外部サービス接続で得た手順は、README または docs に再現可能な形で残す。
- issue 一覧は Status と見出しが矛盾しないよう、定期的に整理する。

## Codex の作業姿勢

- 既存の設計、命名、ファイル構成を尊重する。
- 変更は issue の範囲に収める。
- 実装前に必要な仕様を明確にし、会話上で短く共有してから着手する。
- 実装後に検証結果を共有する。
- ユーザーの未コミット変更を勝手に戻さない。
- 不明点があっても合理的に仮定できる場合は前に進め、リスクが高い場合だけ確認する。
